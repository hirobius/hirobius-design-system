## Framework

For each gate in the inventory, classify the debt class:

- **A — fix-all-now**: small surface, ship the fixes this week. Action-oriented; A means a real commit lands.
- **B — frozen + ratchet**: freeze the count in a baseline file AND author a CI ratchet that asserts the count strictly decreases on each commit. **A row classified B without a ratchet is theater** — convert to D (gate too strict) or C (accept as legacy) instead. Per Adrian 2026-05-06: "yes ratchets" — every B-class closure ships a working ratchet as part of `13p-6`.
- **C — bypass-with-reason**: documented exception in `registry.json` (`pureExceptions[<gate-id>] = { reason, allowedFiles }`). One-time work; no ongoing decrement to drift on.
- **D — re-classify-the-gate**: gate is too strict / too broad / firing on the wrong cohort. Narrow scan scope, downgrade severity to `warn`, move `firingChannel` to `pnpm-meta`, or delete the entry entirely with rationale.

> **Anti-aspirational rule:** any B classification must have a named ratchet script. If we can't write the ratchet, the row isn't B. Drift detected via `pnpm closure:plan` regen + `13p-6` audit which checks that every B row's named ratchet exists and is wired into a strict channel.

Workflow:

1. Adrian fills `classification` for each row interactively (`13p-5-classify-debt-class-batch1`).
2. Each classification gets a one-line rationale on the same row.
3. `13p-6-execute-closures-batch1` walks rows in order (smallest blast radius first) and ships per-closure commits.
4. Re-running `node scripts/run-gates.mjs --channel ci-pr --emit-inventory /tmp/post.json` after the batch must show `totalsByExitCode["1"]` strictly less than the baseline.
